# 提示词优化对比实验

## 实验主题
创建一个 **Leaflet 地图测距工具组件**

---

## 一、提示词对比

### 1.1 粗略提示词
```
创建一个地图测量组件
```

**特点：**
- 只有核心需求，无细节说明
- 没有技术栈指定
- 没有功能边界定义
- 没有用户体验要求

---

### 1.2 精细提示词
```
请创建一个 Vue 3 + TypeScript + Leaflet 的地图测距工具组件，要求：

【技术要求】
- 使用 <script setup> 语法
- TypeScript 类型安全，定义完整的接口
- 使用 shallowRef 优化地图实例性能
- 组件可通过 Props 配置（中心点、缩放、底图、颜色等）

【功能需求】
1. 点击开始/结束测量按钮切换状态
2. 测量时点击地图添加测量点
3. 实时显示鼠标跟随的临时距离预览
4. 双击快速结束测量
5. 支持距离单位自动转换（米/公里）
6. 显示已添加的点数量
7. 可一键清除所有测量

【UI/UX 要求】
- 测量状态下光标变为十字准星
- 标记点使用自定义样式（彩色圆点带阴影）
- 测量线段有合适的宽度和透明度
- 按钮有激活状态样式
- 有操作提示信息
- 整体布局美观，使用 scoped 样式

【工程化要求】
- 完整的错误处理（try-catch）
- 组件卸载时完整清理事件监听器和图层
- 通过 defineExpose 暴露必要方法给父组件
- 代码有清晰的注释
```

**特点：**
- 明确技术栈和语法规范
- 功能点逐条列出
- 包含 UI/UX 细节要求
- 有工程化和质量标准

---

## 四、核心改进点总结

### 4.1 精细提示词带来的提升
1. **明确技术选型** → 直接生成符合项目栈的代码
2. **细化功能边界** → 避免功能遗漏或过度设计
3. **强调工程化** → 代码更健壮、可维护
4. **关注用户体验** → 产品级而非 Demo 级
5. **制定质量标准** → 有明确的验收依据

### 4.2 提示词优化原则
| 原则 | 说明 |
|------|------|
| **具体化** | 避免"创建一个组件"，而是"创建一个X功能的Y组件" |
| **技术栈明确** | 明确框架、版本、语法规范 |
| **功能清单化** | 逐条列出期望功能点 |
| **质量标准** | 包含错误处理、类型安全、性能等要求 |
| **UI/UX 细节** | 描述交互方式、视觉风格 |

## 六、结论

**精细提示词投资回报率极高！**

- 虽然提示词编写时间增加 ~3-5 倍
- 但代码质量和功能完整性提升 ~10 倍以上
- 大幅减少后续修改和调试成本
- 更符合生产环境要求

**最佳实践：**
> 写提示词时，把自己当作产品经理 + 技术负责人，把需求、技术、质量都讲清楚！
